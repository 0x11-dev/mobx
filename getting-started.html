MobX

Ten minute introduction to MobX and React

MobX is a simple, scalable and battle tested state management solution. This introduction will teach you all the imporant concepts of MobX in ten minutes.

Core idea

State is at the heart of each application and there is no quicker way to create buggy, unmanagable applications then by producing inconsistent states. Hence many state management solutions try to restrict the ways in which you can modify state, for example by making state immutable. But this introduces new problems; data need to be normalizeds, referential integrity can no longer be guaranteed and it becomes next to impossible to use convenient concepts like prototypes.

MobX makes state management simple again by addressing the root issue: it makes it impossible to produce inconsistent states. The strategy to achieve this is simple: Make sure that everything that can be derived from the application state, will be derived. Automatically.

Conceptually MobX treats your application like a spreadsheet.

  1. First of all, there is the application state. Graphs of objects, arrays, primitives, references that forms the model of your application. These values are the “data cells” of your application. Enterprise grade applications have complex, highly connected data models and MobX supports this.
  2. MobX helps you managing all the data that describes the state of your application. Data can have many forms, objects, arrays, references, class instances. Unlike many other state management libraries, MobX doesn't care. If you were creating a spreadsheet; the values of your data cells together are your application state. If you are building a todo application, your state is the list of todos.
  3. Secondly there are derivations. Basically, and this is the core idea of MobX: Any value that can be computed automatically from the state of your application, should be (re-)computed automatically. MobX will take care of this, as long as you provide a derivation. These derivations, or computed values, can range from be simple values like the number of unfinished todos, to complex stuff like a visual HTML representation of your todos. In spreadsheet terms: these are the formulas and charts of your application.
  4. Reactions are very similar to derivations. The main difference: they don't produce a value, but they execute some task for you whenever relevant state is changed. They make sure that network requests are made automatically at the right time, or that debug information is printed upon each change.
  5. Finally there are actions. Basically; all other code. Actions live outside MobX but might alter your state. MobX will make sure that any changes to the application state caused by your actions are automatically processed by all derivations and reactions. Syncronously and glitch-free.

Making react components reactive

Ok, so far we made a silly report reactive. Let's build a reactive user interface around this very same store. React components are (despite their name) not reactive out of the box, you need to explicitly call setState or re-instantiate them in the parent's render method to force them to re-render after data has been changed.

To avoid boilerplate and to make sure components update smartly, the mobx-react package provides the observer decorator / function that turns a component into a reaction. From there on, you never have to worry about when to update your components! It's conceptually no different from auto-running the report function.

The listing below defines some plain old react components. The only MobX thing in there is that they are passed through the observer to make them reactive. After that, each component will individually re-render when data changes. You don't have to call setState anymore, nor do you have to figure out how subscribe to the proper parts of your application state using selectors or higher order components that need to be configured. Basically, all components have become smart but are still defined in a dumb, declarative manner.

Asynchronous actions

Since everything in our small Todo application is derived from the state, it really doesn't matter when state is changed. That makes creating asynchronous actions really straight forward. Just press the the following button (multiple times) to asynchronously emulate loading new todo items:

The code behind that is really straight forward. We start with uppdating the store property pendingRequests to have the UI reflect the current loading status. Once loading is finished, we update the todo's of the store and decrease the pendingRequests counter again. A counter is used to make sure we handle with multiple parallel requests properly: 

observableTodoStore.pendingRequests++;
setTimeout(function() {
    observableTodoStore.addTodo('Random Todo ' + Math.random());
    observableTodoStore.pendingRequests--;
}, 2000);

Conclusion

That's all! No boilerplate. Just some simple, declarative components that form our complete UI. And which are derived completely, reactively from our state. You are now ready to start using the mobx and mobx-react packages in your own applications. A short summary of the things you learned so far:

  1. Use observable(object or array) or extendObservable(object, propertiesMap) to make objects suitable for using them in derivations.
  2. Passing argumentless functions to extendObservable allows you to create “formulas”. Functions that compute a value from the current state and which will be recomputed automatically (if observed).
  3. Use autorun to automatically run functions that depend on some observable state. This is useful for logging, making network requests etc.
  4. Use the observer(react component) function from the mobx-react package to make your React components truly reactive. They will update automatically and efficiently. Even when used in large complex applications with large amounts of data.

Feel free to play around a bit longer with the editable code blocks above to get a basic feeling how MobX reacts to all your changes. You could for example add a log statement to the report function to see when it is called. Or don't show the report at all and see how that influences the rendering of the TodoList. Or show it only under specific circumstances...
